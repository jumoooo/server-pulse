## ⚠️ 선행 해결 항목 (코드 착수 전 필수)

Step 3 진입 전에 반드시 정리해야 하는 불일치.

### 타입 정의 불일치

기획서와 실제 코드(`src/types/server.ts`)의 도메인 모델이 다르다.

| 항목      | 기획서                                                 | 현재 코드               | 할 일                   |
| --------- | ------------------------------------------------------ | ----------------------- | ----------------------- |
| 서버 상태 | `"healthy" \| "degraded" \| "down"`                    | `"online" \| "offline"` | 기획서 기준으로 통일    |
| 서버 지표 | CPU, 메모리, RTT, 에러율, 플레이어 수                  | latencyMs만 존재        | `MetricPoint` 타입 추가 |
| 알림 모델 | severity, status(`open/acknowledged/resolved`), ruleId | 기본 구조만 존재        | status 필드 정합 확인   |
| 업타임    | `uptimeSeconds`, `lastHeartbeatAt`                     | 없음                    | 타입에 추가             |

> **권장**: `src/types/server.ts` 한 곳에서 기획서 기준으로 정리 후 Mock 데이터도 함께 수정.

### API Mock 누락

현재 `/api/servers/route.ts`만 존재. 기획서에 필요한 나머지 Mock이 없다.

| 필요한 엔드포인트               | 현재                            |
| ------------------------------- | ------------------------------- |
| `GET /api/servers`              | ✅ 있음 (3개 서버)              |
| `GET /api/servers/[id]`         | ❌ 없음                         |
| `GET /api/servers/[id]/metrics` | ❌ 없음 (MetricPoint 시계열)    |
| `GET /api/alerts`               | ❌ 없음                         |
| `PATCH /api/alerts/[id]`        | ❌ 없음 (ack/resolve 상태 변경) |

## Step 3 — 대시보드 구현

> **관련 파일**: `src/app/(main)/dashboard/page.tsx`, `src/components/dashboard/`

현재 컴포넌트 3개(`ServerStatusCard`, `MetricsSummary`, `PlayerChart`) 모두 `return null`.

### 작업 목록

- [ ] **`ServerStatusCard`** — 온라인 / 주의 / 장애 서버 수 카드 (3종)
- [ ] **`MetricsSummary`** — 전체 평균 RTT, 평균 에러율, 총 플레이어 수 수치 카드
- [ ] **`PlayerChart`** — Recharts `LineChart`로 24시간 플레이어 수 추이
- [ ] **대시보드 페이지** — 위 3개 컴포넌트 조합 + Loading/Error/Empty state 적용
- [ ] **"문제 서버 Top N"** — 알림이 많거나 status가 degraded/down인 서버 목록 (기획서 5-IA)

### 기술 참고

- `PlayerChart`는 `'use client'` 필요 (Recharts는 CSR 전용)
- 데이터는 Step 4 훅이 완성된 후 연결, 그 전까지는 Mock 배열 직접 사용
- 차트 데이터는 `MetricPoint[]` 타입의 시계열 배열 형태

## Step 4 — TanStack Query 훅 구현

> **관련 파일**: `src/hooks/useServers.ts`, `src/hooks/useServerDetail.ts`

현재 두 훅 모두 빈 객체 반환하는 가짜 구현.

### 작업 목록

- [ ] **`useServers()`** — `GET /api/servers` 호출 + `refetchInterval: 10_000` 폴링
- [ ] **`useServerDetail(id)`** — `GET /api/servers/[id]` 호출 + 폴링
- [ ] **`useServerMetrics(id)`** — `GET /api/servers/[id]/metrics` 호출 (신규 훅 필요)
- [ ] **`useAlerts()`** — `GET /api/alerts` 호출 (신규 훅 필요)
- [ ] **`useAlertMutation()`** — `PATCH /api/alerts/[id]` 상태 변경 (신규 훅 필요)
- [ ] 로딩 / 에러 / 빈 상태 반환값 표준화

### 기술 참고

```ts
// 폴링 패턴 (기획서 10초 명시)
useQuery({
  queryKey: ["servers"],
  queryFn: () => fetchApi<Server[]>("/servers"),
  refetchInterval: 10_000,
  staleTime: 5_000,
});
```

## Step 5 — 서버 목록 / 상세 페이지

> **관련 파일**: `src/app/(main)/servers/`, `src/components/servers/`

현재 `ServerList`, `ServerDetail` 모두 `return null`. 페이지도 placeholder.

### 서버 목록 (`/servers`)

- [ ] **`ServerList`** — 서버 카드 그리드 or 테이블 (상태, 지역, 버전, 플레이어 수, 최근 알림 수 표시)
- [ ] **필터** — 상태(`healthy/degraded/down`), 지역 드롭다운
- [ ] **검색** — 이름/ID 텍스트 검색
- [ ] 서버 클릭 시 상세 페이지(`/servers/[id]`)로 이동
- [ ] `useServers()` 훅 연결 + Loading/Error/Empty state

### 서버 상세 (`/servers/[id]`)

- [ ] **메타 요약 카드** — 업타임, 버전, 마지막 heartbeat, 지역
- [ ] **지표 차트 (Recharts)** — 플레이어 수 / CPU / 메모리 / RTT / 에러율 (탭 또는 섹션 구분)
- [ ] **기간 선택** — 1h / 6h / 24h 토글
- [ ] **관련 알림 리스트** — 해당 서버 기준 최근 N개
- [ ] params 처리 (`const { id } = await params` — Next.js 15 방식)
- [ ] `useServerDetail(id)` + `useServerMetrics(id)` 훅 연결

## Step 6 — 알림 페이지

> **관련 파일**: `src/app/(main)/alerts/page.tsx`, `src/components/alerts/AlertList.tsx`

현재 `AlertList`는 `return null`. 페이지는 placeholder.

### 작업 목록

- [ ] **`AlertList`** — 알림 카드/행 (severity 배지, status, 서버명, 발생 시각, 규칙/설명)
- [ ] **필터** — severity(`critical/warning/info`), status(`open/acknowledged/resolved`), 서버 선택
- [ ] **상태 변경 UI** — ack / resolved 버튼 (백엔드 없으면 Zustand로 낙관적 업데이트)
- [ ] **알림 → 서버 상세 링크** — 기획서 "1클릭 이동" 원칙
- [ ] `useAlerts()` + `useAlertMutation()` 훅 연결 + Loading/Error/Empty state

### 기술 참고

- 상태 변경은 `useOptimistic` (React 19) 또는 TanStack Query `useMutation` + optimistic update 선택
- severity 색상: `critical` → red, `warning` → yellow, `info` → blue

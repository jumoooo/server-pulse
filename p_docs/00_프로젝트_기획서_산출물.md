# ServerPulse 프로젝트 기획서 산출물 📄

> Next.js 15(App Router) + React 19 + TypeScript + Tailwind 기반  
> 목적: **게임 서버 Observability(관측) + 알림 + (향후) AI 인사이트** 제공

---

## 0) 프로젝트 초기 세팅 및 전체 구현 계획 🧱🗺️

### 배경
- **기존 MCP 서버를 백엔드로 활용**: `steam-game-server-mcp` (`https://github.com/jumoooo/steam-game-server-mcp`)
- 목표: Next.js 기반 **실시간(폴링 기반) 모니터링 대시보드** 구현
- 프로젝트명/설명: **ServerPulse — AI-powered game server observability platform**

### 1단계: 셋업(기술 스택 고정)
- **Framework**: Next.js 15 (App Router)
- **Language**: TypeScript (strict)
- **Styling**: Tailwind CSS
- **State(UI)**: Zustand
- **Server State**: TanStack Query (React Query v5)
- **Charts**: Recharts
- **Package Manager**: npm
- **Tooling**: ESLint, Prettier

> 참고: 현재 워크스페이스는 위 스택이 이미 설치/구성되어 있으며, `/` → `/dashboard` 리다이렉트도 구현되어 있다.

#### 현재 코드와의 차이(정합 메모)
- 현재는 라우트가 `src/app/(main)/dashboard`, `src/app/(main)/servers`, `src/app/(main)/alerts` 형태로 잡혀 있음(라우트 그룹 사용).
- 문서의 “목표 구조”는 라우트 그룹 없이 단순 구조로 표기했으므로, 필요 시 **(main) 라우트 그룹 유지 vs 제거**를 먼저 결정하고 폴더를 정리한다.

### 3단계: 작업 순서(구현 Step)

#### Step 1 — 레이아웃 구성
- Sidebar, Header 컴포넌트 구현
- 반응형 레이아웃(모바일 고려)
- 다크 톤 또는 SaaS 느낌의 디자인

#### Step 2 — API Route 구현
- `src/app/api/servers/route.ts`에서 **MCP 서버 응답을 모킹 데이터로 먼저 구현**
- 이후 실제 MCP 연결로 교체할 수 있도록 구조를 추상화

#### Step 3 — 대시보드 페이지
- `ServerStatusCard`: 온라인/오프라인 서버 수
- `MetricsSummary`: 전체 플레이어 수, 평균 레이턴시
- `PlayerChart`: Recharts `LineChart`로 플레이어 수 추이

#### Step 4 — TanStack Query 폴링
- `useServers` hook: 10초마다 서버 상태 자동 갱신
- 로딩/에러/빈 상태 처리 포함

#### Step 5 — 서버 목록 및 상세 페이지
- `/servers`: 서버 카드 리스트(상태, 플레이어 수, ping)
- `/servers/:id`: 플레이어 목록, 레이턴시 히스토리 차트

#### Step 6 — 알림 페이지
- `/alerts`: 서버 다운, 레이턴시 급등 이벤트 목록
- 알림 심각도(`critical / warning / info`) 구분

#### Step 7 — 완성도 작업
- 모든 페이지 Loading, Error, Empty state 구현(표준화)
- `README.md` 작성(아키텍처 다이어그램 포함)

#### Step 8 — AI 분석(마지막)
- `/servers/:id` 상세 페이지에 `Analyze` 버튼 추가
- 버튼 클릭 시 서버 상태 데이터를 **Anthropic API**로 전달
- 분석 결과(원인 추정 + 권장 조치)를 **스트리밍**으로 표시

### 주의사항(품질 게이트)
- **Server Component / Client Component 역할 분리**
- `'use client'` 최소화(상태/이벤트/차트처럼 필요한 곳만)
- 타입은 `any` 금지, `src/types/server.ts`에 정의 후 사용
- 각 페이지는 **Loading / Error / Empty state**를 반드시 포함

### 커밋 전략 메모(중요)
- 요청사항은 “Step별 커밋 분리”이지만, **현재 프로젝트는 git repo가 아님**  
  → Step별 커밋을 하려면 먼저 `git init` 및 `.gitignore` 정리가 필요하다.

## 1) 한 줄 요약 🧭

**ServerPulse**는 여러 게임 서버(또는 게임 서버 클러스터)의 상태/성능 지표를 한 곳에서 모니터링하고, 이상 징후를 알림으로 전달하며, 운영자가 빠르게 원인을 좁히도록 돕는 관측(Observability) 대시보드다.

---

## 2) 문제 정의 & 성공 기준 🎯

### 문제 정의
- 운영자는 서버 상태를 확인하기 위해 여러 툴/로그를 오가며 **상황 파악 시간이 길어짐**
- 장애는 “이미 터진 후” 알게 되는 경우가 많고, **탐지(Detect) → 대응(Respond)** 이 늦음
- 지표가 많아도 “그래서 지금 뭘 해야 하지?”로 연결되지 않음

### 성공 기준(KPI/OKR 예시)
- **MTTD(탐지 시간)**: 알림으로 5분 이내 탐지
- **MTTR(복구 시간)**: 장애 원인 후보를 10분 내 3개 이내로 좁힘
- **운영 효율**: 일일 점검 시간을 30% 단축
- **알림 신뢰도**: 오탐(False Positive) 비율 10% 이하

---

## 3) 타겟 사용자 & 사용 시나리오 👤

### 핵심 사용자
- **게임 서버 운영자(SRE/DevOps/서버 엔지니어)**
- **라이브 서비스 담당 개발자(백엔드/인프라)**
- (확장) **프로듀서/운영 PM**: 고수준 상태(가용성/플레이어)만 확인

### 대표 시나리오
- **일상 모니터링**: 대시보드에서 전체 서버의 상태 요약을 보고 “문제 서버”를 즉시 찾는다.
- **이상 감지**: 특정 서버의 CPU/메모리/RTT 급증 → 알림 확인 → 서버 상세로 이동해 원인을 좁힌다.
- **알림 대응**: 알림 상세에서 증상/최근 변화/관련 지표를 확인하고 조치(재시작, 스케일링, 롤백)를 수행한다.

## 4) 제품 범위(스코프) ✅

### MVP (현재 라우팅 구조 기준)
현재 코드베이스 라우트가 이미 잡혀 있음:
- `/dashboard` 대시보드 (Step 3)
- `/servers` 서버 목록 (Step 5)
- `/servers/[id]` 서버 상세 (Step 5)
- `/alerts` 알림 목록/상세 (Step 6)

MVP에서 반드시 제공할 기능:
- **대시보드**
  - 전체 서버 상태 요약(정상/주의/장애)
  - 핵심 지표 스냅샷(예: 평균 RTT, 에러율, CPU/메모리, 플레이어 수)
  - 최근 24h 추세(간단 차트)
- **서버 목록**
  - 서버 카드/테이블(상태, 지역, 버전, 현재 플레이어, 최근 알림 수)
  - 검색/필터(상태/지역/버전)
- **서버 상세**
  - 서버 메타(이름/ID/지역/버전/업타임)
  - 지표 타임라인(플레이어/CPU/메모리/RTT/에러율)
  - 최근 알림 리스트(서버 스코프)
- **알림**
  - 알림 리스트(심각도/발생 시각/대상 서버/규칙)
  - 알림 상태(신규/확인/해결)

### 후순위(향후 확장)
- AI 인사이트(원인 후보/변화점 탐지/요약 리포트)
- 알림 규칙 관리 UI(임계값, SLO 기반)
- 사용자/조직, 권한(RBAC)
- 다중 프로젝트/환경(prod/stage) 분리
- 온콜 연동(Slack, Discord, PagerDuty 등)

## 5) 정보 구조(IA) & 네비게이션 🗺️

### IA(초안)

## 6) 핵심 데이터(도메인) 모델 🧩

> MVP에서는 백엔드가 없어도 더미 데이터로 UI/UX를 확정할 수 있도록 도메인 모델을 먼저 고정한다.

### Server
- `id`: string
- `name`: string
- `region`: string (예: `kr-seoul`, `us-east`)
- `version`: string (게임 서버 빌드/배포 버전)
- `status`: `"healthy" | "degraded" | "down"`
- `uptimeSeconds`: number
- `lastHeartbeatAt`: ISO string

### MetricPoint (타임시리즈)
- `timestamp`: ISO string
- `cpuUsage`: number (0~100)
- `memoryUsage`: number (0~100)
- `rttMs`: number
- `errorRate`: number (0~1 또는 %)
- `playerCount`: number

### Alert
- `id`: string
- `serverId`: string
- `severity`: `"info" | "warning" | "critical"`
- `status`: `"open" | "acknowledged" | "resolved"`
- `title`: string
- `description`: string
- `ruleId`: string
- `createdAt`: ISO string
- `resolvedAt?`: ISO string

## 7) 지표 정의(초안) 📈

### 운영 지표(인프라/서비스)
- **CPU 사용률(%)**: 스파이크/지속 상승 감지
- **메모리 사용률(%)**: 누수/GC pressure 징후
- **RTT(ms)/Latency**: 네트워크/부하 징후
- **에러율(%)**: 5xx, 매치메이킹 실패 등 도메인 에러 포함 가능
- **업타임/Heartbeat**: 다운/네트워크 단절 감지

### 게임 도메인 지표(서비스 품질)
- **동시 접속자(playerCount)**: 수요/세션 상태
- (확장) 매치 성공률, 평균 매치 시간, 틱레이트, 패킷 드랍률 등

## 8) 알림 정책(초안) 🔔

### 알림 레벨
- **info**: 참고(추적) 수준, 대시보드 배너/리스트만
- **warning**: 주의, 운영자 확인 필요
- **critical**: 장애/긴급, 즉시 대응 필요

### MVP 규칙 예시
- **critical**: Heartbeat 2분 이상 없음(서버 다운 가능성)
- **warning**: RTT 300ms 이상이 5분 지속
- **warning**: 에러율 3% 이상이 10분 지속
- **info**: 배포 직후 30분간 에러율 변동 증가(관찰용)

# Demo/Prod 브랜치 분리 병렬 실행 상세 계획서

## Internal Instructions (English)

### 1) Executive Summary

The project will be operated in two stable lanes:

- `demo`: publicly accessible mock-data portfolio lane
- `main`: production lane integrated with `steam_mcp`

Execution strategy:

- Perform governance and branch/environment decisions sequentially first.
- Run API integration, test architecture, and deployment pipeline setup in parallel.
- Enforce quality gates before each promotion step.
- Maintain rollback readiness at code, config, and deployment routing levels.

### 2) Workstreams

| ID | Purpose | Parallel | Prerequisite | Deliverable |
| --- | --- | --- | --- | --- |
| WS-00 | Governance baseline | No | None | Policy/RFC |
| WS-01 | Branch strategy (`main`/`demo`) | No | WS-00 | Branch protection + merge policy |
| WS-02 | Environment split | Partial | WS-00 | Env matrix and secret scope |
| WS-03 | Data provider abstraction | Yes | WS-02 | Provider contracts |

| WS-08 | Cutover rehearsal | No | WS-04~WS-07 | Go/No-Go report |
| WS-09 | Hypercare stabilization | No | WS-08 | Post-cutover report |

### 3) Phase Plan

#### Phase 0 (Sequential): Governance and Scope Lock
- Confirm branch responsibility (`main=prod`, `demo=portfolio`).
- Assign approval roles and incident ownership.
- Define SLO and deployment gate thresholds.

#### Phase 1 (Sequential + Partial Parallel): Foundation Split
- Sequential:
  - Apply branch protection and merge policy.
  - Define environment keys (e.g., `DEMO_MODE`, `STEAM_MCP_*`).
- Parallel:
  - Separate Vercel deployment targets.
  - Separate secret scopes (no production secrets in demo).
  - Draft runbooks.

#### Phase 2 (Parallel-first): Architecture Split
- Parallel:
  - Define `DataSource` contract.
  - Keep `MockDataProvider` for `demo`.
  - Implement `SteamMcpProvider` for `main` with timeout/retry/error mapping.
- Sequential:
  - Apply provider injection at route handlers.
  - Finalize failure behavior for each lane.

#### Phase 3 (Parallel-first): Quality System
- Parallel:
  - Provider contract tests.
  - Lane-specific E2E smoke tests.
  - Lane-specific CI workflows.
  - Runtime observability and alerting.
- Sequential:
  - Bind all checks into deployment gate.

#### Phase 4 (Sequential): Rehearsal and Cutover
- Preview rehearsal for both lanes.
- Staging rehearsal with failure injection (`steam_mcp` latency/failure).
- Go/No-Go decision and production promotion.

#### Phase 5 (Sequential): Hypercare
- 24-72h monitoring window.
- Trigger rollback if thresholds are exceeded.
- Record lessons learned and lock operating policy.

### 4) Stability Design (Risk Register)

| Risk | Prevention | Detection | Mitigation |
| --- | --- | --- | --- |
| Demo code leaks into main | Protected branches + CODEOWNERS | PR checks for mock imports | Immediate revert/hotfix |
| Secret leakage to demo | Env-scoped secret policies | Secret audit pre-deploy | Token revocation and redeploy |
| Contract mismatch | Schema validation and contracts | Contract tests + runtime validation logs | Mapper patch |
| `steam_mcp` outage/latency | Timeout/retry/circuit strategy | Error-rate + latency alerts | Degraded mode and fallback UX |
| Post-switch performance drop | Caching and request optimization | p95/p99 dashboards | Tune cache and isolate heavy routes |
| Demo-only tests pass, prod fails | Dual-lane mandatory tests | Gate-level lane checks | Block release and triage |
| Wrong branch deployed | Branch-environment binding | Deployment metadata precheck | Stop and redeploy previous artifact |
| Slow incident response | Runbook and ownership table | MTTA/MTTR in drills | Incident template and escalation |
| Data integrity gaps | Required-field validation | Parse/validation failure metrics | Graceful fallback and patch |

### 5) Validation and Deployment Gates

- Gate 1: Static checks (`lint`, `type-check`) pass
- Gate 2: Lane-specific required tests pass
- Gate 3: Observability dashboard and alerts active
- Gate 4: Rollback procedure validated
- Gate 5: Minimum 2 approvals for promotion

Quick gate commands:

- 🚦 `pnpm gate:lane:demo`
- 🚦 `pnpm gate:lane:prod`
- 🛑 `prod` must fail with non-zero exit when `DEMO_MODE=true` or no configured `STEAM_MCP_*` key exists.

Acceptance baseline:

- `demo` runs without production secrets.
- `main` serves core flows via `steam_mcp`.
- Data-provider split is explicit and auditable.
- Branch-to-environment mapping prevents accidental release.

### 6) Rollback Strategy

- Code rollback to last stable tag.
- Config rollback via feature flags/environment routing.
- Deployment alias rollback to previous stable deployment.
- Incident-close requirements:
  - root-cause summary
  - mitigation record
  - prevention action item

## 사용자용 설명 (한글)

### 1) 요약

프로젝트를 두 개의 운영 라인으로 분리합니다.

- `demo`: 채용팀 공개용, 목데이터 기반 데모 라인
- `main`: `steam_mcp` 실서버 연동 실전 라인

진행 방식:

- 먼저 정책/브랜치/환경을 순차로 고정
- 이후 API 연동/테스트/배포 파이프라인을 병렬 수행
- 각 단계에서 배포 게이트 통과 시에만 다음 단계로 승격
- 코드/설정/배포 라우팅 3단 롤백 준비

### 3) 단계별 실행 계획

#### Phase 0 (순차): 정책/범위 고정
- `main=실전`, `demo=포폴` 책임을 확정
- 승인자/장애 대응 책임자 지정
- SLO/배포 게이트 기준 확정

#### Phase 1 (순차 + 부분 병렬): 분리 기반
- 순차:
  - 브랜치 보호 규칙 적용
  - 환경 변수 체계 정의 (`DEMO_MODE`, `STEAM_MCP_*`)
- 병렬:
  - Vercel 배포 타깃 분리
  - 시크릿 스코프 분리
  - 런북 초안 작성

#### Phase 2 (병렬 중심): 아키텍처 분리
- 병렬:
  - `DataSource` 계약 정의
  - `demo`용 `MockDataProvider` 유지
  - `main`용 `SteamMcpProvider` 구현(타임아웃/재시도/에러 매핑)
- 순차:
  - Route Handler 주입 방식 통일
  - 라인별 장애 동작 확정

#### Phase 3 (병렬 중심): 품질 체계
- 병렬:
  - Provider 계약 테스트
  - 레인별 E2E 스모크
  - 레인별 CI 워크플로
  - 런타임 관측/알림
- 순차:
  - 배포 게이트 통합

#### Phase 4 (순차): 리허설/컷오버
- Preview 리허설
- Staging 장애 주입 테스트
- Go/No-Go 승인 후 승격

#### Phase 5 (순차): 하이퍼케어
- 24~72시간 집중 모니터링
- 임계치 초과 시 즉시 롤백
- 회고 후 정책 고정

### 4) 안정성 설계 (핵심 리스크)

| 리스크 | 예방 | 탐지 | 완화 |
| --- | --- | --- | --- |
| 데모 코드가 main 유입 | 브랜치 보호/CODEOWNERS | PR에서 mock import 점검 | 즉시 revert |
| 데모에 실시크릿 유출 | 환경 스코프 분리 | 배포 전 시크릿 감사 | 토큰 폐기/재발급 |
| 응답 스키마 불일치 | 계약/스키마 검증 | 계약 테스트/런타임 검증 로그 | 매퍼 핫픽스 |
| `steam_mcp` 장애 | 타임아웃/재시도 | 실패율/지연 알림 | degraded mode |
| 성능 저하 | 캐시/요청 최적화 | p95/p99 추적 | 고비용 경로 완화 |
| 테스트 사각지대 | 레인별 필수 테스트 | 게이트 강제 | 배포 차단 후 수정 |
| 브랜치 오배포 | 브랜치-환경 바인딩 | 배포 메타 사전 검사 | 즉시 재배포 |
| 장애 대응 지연 | 런북/담당자 지정 | 리허설 MTTA/MTTR | 에스컬레이션 체계 |
| 데이터 정합성 문제 | 필수 필드 검증 | 파싱 실패율 알림 | 안전 기본값 적용 |

### 5) 검증 및 배포 게이트

- Gate 1: `lint`, `type-check` 통과
- Gate 2: 레인별 필수 테스트 통과
- Gate 3: 관측/알림 활성 확인
- Gate 4: 롤백 절차 검증
- Gate 5: 2인 이상 승인

빠른 게이트 명령:

- 🚦 `pnpm gate:lane:demo`
- 🚦 `pnpm gate:lane:prod`
- 🛑 `prod`는 `DEMO_MODE=true` 이거나 설정된 `STEAM_MCP_*` 키가 없으면 **반드시 실패(비정상 종료 코드)** 해야 함

수용 기준:

- `demo`는 실서버 시크릿 없이 안정 동작
- `main`은 `steam_mcp` 실데이터로 핵심 플로우 동작
- 데이터 공급 경로가 코드상 명확히 분리
- 브랜치-환경 매핑 오류가 게이트에서 차단
